# Org Scope And Sprint Review

Date: 2026-03-10
Purpose: check whether the current org plan is deep enough for the owner-led pyramid model and define the next execution sequence.

## Straight Verdict

The planning is now strong enough to keep building on.

It is not yet complete enough in runtime to say Clubroom already functions like `Johnny's Coaching LTD`.

The current planning stack is good on:

- one-org model
- hierarchy
- commercial ownership
- visibility rules
- market baseline

The current planning stack was not deep enough on:

- owner-controlled commercial settings in the real runtime surface
- staff assignment and reassignment as a first-class operating console
- head coach oversight and tasking
- coach split between org work and independent work
- family support ownership and trust copy

So the answer is:

- the direction is right
- the rules are mostly right
- the next sprint order needed tightening around real org operations

## What Is Locked

These rules should now be treated as fixed unless a later macro decision reopens them:

1. One org model.
2. `academy` is not a separate runtime architecture.
3. The org owner chooses the org commercial mode.
4. V1 commercial mode is org-level, not per-session.
5. V1 commercial mode changes are prospective only.
6. Coaches can belong to multiple orgs and still run independent work.
7. Coach marketplace relationship language is:
   - `Follow Coach`
   - `Save Coach`
   - `Request Contact`
8. Current money truth is still reconciler plus off-app/direct payment guidance, not real payout rails.

## What Already Exists In Runtime

The repo already has useful foundations:

- `actingAs`, `clubId`, `ownerCoachId`, `assigneeCoachId`
- club settings and membership
- coach manage surfaces
- booking review and confirmation relationship copy
- club-owned session creation and assignee selection

The repo does not yet have:

- owner commercial controls in the live settings path
- a real owner/admin staffing console
- a real head coach operating surface
- a coach view split cleanly into org work vs independent work
- family-facing support ownership and visibility language across the whole booking lifecycle

## Does It Function Like The Desired Pyramid Yet

Not yet in runtime.

It does in planning, if the next sprints are built in the right order.

The product will start to match the desired pyramid only when all of the following are true:

1. the owner can set commercial mode in the real org settings surface
2. owner/admin can assign and reassign staff from an org console
3. head coach has a distinct oversight and standards surface
4. coaches can clearly separate org assignments from their own direct business
5. families can see who they booked with, who delivers, who bills, and who handles problems

## Why The Old Follow-On Sprint Order Was Not Good Enough

The earlier sprint stack still jumped too quickly into broad themes such as relationship cleanup, coach realism, and lifecycle honesty without first finishing the core org operating loop.

That was the wrong emphasis for the product you described.

The owner-led org model needs these questions answered first:

1. who owns the money
2. who controls staff work
3. who supervises coaches
4. how org work differs from coach-owned work
5. what parents see when responsibility changes

## Revised Sprint Order

1. `PR-03` Commercial Mode And Booking Truth
2. `PR-04` Org Assignment And Staffing Console
3. `PR-05` Head Coach Oversight And Tasking
4. `PR-06` Coach Work Split And Earnings Realism
5. `PR-07` Family Trust And Support Ownership
6. `PR-08` Relationship Model And Blocking Integrity
7. `PR-09` Account, Auth, And Admin Ops Honesty
8. `PR-10` Org Surface Migration And Naming Cleanup

## Principle For The Next Sprints

Do not build more org language without building the operating surfaces behind it.

Do not build more money language without preserving current finance honesty.

Do not build more role names without giving those roles distinct responsibilities in product.
